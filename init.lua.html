 23 What is Kickstart?
 24 
 25   Kickstart.nvim is *not* a distribution.
 26 
 27   Kickstart.nvim is a starting point for your own configuration.
 28     The goal is that you can read every line of code, top-to-bottom, understand
 29     what your configuration is doing, and modify it to suit your needs.
 30 
 31     Once you've done that, you can start exploring, configuring and tinkering to
 32     make Neovim your own! That might mean leaving Kickstart just the way it is for a while
 33     or immediately breaking it into modular pieces. It's up to you!
 34 
 35     If you don't know anything about Lua, I recommend taking some time to read through
 36     a guide. One possible example which will only take 10-15 minutes:
 37       - https://learnxinyminutes.com/docs/lua/
 38 
 39     After understanding a bit more about Lua, you can use `:help lua-guide` as a
 40     reference for how Neovim integrates Lua.
 41     - :help lua-guide
 42     - (or HTML version): https://neovim.io/doc/user/lua-guide.html
 43 
 44 Kickstart Guide:
 45 
 46   TODO: The very first thing you should do is to run the command `:Tutor` in Neovim.
 47 
 48     If you don't know what this means, type the following:
 49       - <escape key>
 50       - :
 51       - Tutor
 52       - <enter key>
 53 
 54     (If you already know the Neovim basics, you can skip this step.)
 55 
 56   Once you've completed that, you can continue working through **AND READING** the rest
 57   of the kickstart init.lua.
 58 
 59   Next, run AND READ `:help`.
 60     This will open up a help window with some basic information
 61     about reading, navigating and searching the builtin help documentation.
 62 
 63     This should be the first place you go to look when you're stuck or confused
 64     with something. It's one of my favorite Neovim features.
 65 
 66     MOST IMPORTANTLY, we provide a keymap "<space>sh" to [s]earch the [h]elp documentation,
 67     which is very useful when you're not exactly sure of what you're looking for.
 68 
 69   I have left several `:help X` comments throughout the init.lua

405   { -- LSP Configuration & Plugins

421     config = function()
422       -- Brief aside: **What is LSP?**
423       --
424       -- LSP is an initialism you've probably heard, but might not understand what it is.
425       --
426       -- LSP stands for Language Server Protocol. It's a protocol that helps editors
427       -- and language tooling communicate in a standardized fashion.
428       --
429       -- In general, you have a "server" which is some tool built to understand a particular
430       -- language (such as `gopls`, `lua_ls`, `rust_analyzer`, etc.). These Language Servers
431       -- (sometimes called LSP servers, but that's kind of like ATM Machine) are standalone
432       -- processes that communicate with some "client" - in this case, Neovim!
433       --
434       -- LSP provides Neovim with features like:
435       --  - Go to definition
436       --  - Find references
437       --  - Autocompletion
438       --  - Symbol Search
439       --  - and more!
440       --
441       -- Thus, Language Servers are external tools that must be installed separately from
442       -- Neovim. This is where `mason` and related plugins come into play.
443       --
444       -- If you're wondering about lsp vs treesitter, you can check out the wonderfully
445       -- and elegantly composed help section, `:help lsp-vs-treesitter`
446 
447       --  This function gets run when an LSP attaches to a particular buffer.
448       --    That is to say, every time a new file is opened that is associated with
449       --    an lsp (for example, opening `main.rs` is associated with `rust_analyzer`) this
450       --    function will be executed to configure the current buffer
451       vim.api.nvim_create_autocmd('LspAttach', {
452         group = vim.api.nvim_create_augroup('kickstart-lsp-attach', { clear = true }),
453         callback = function(event)
454           -- NOTE: Remember that Lua is a real programming language, and as such it is possible
455           -- to define small helper and utility functions so you don't have to repeat yourself.
456           --
457           -- In this case, we create a function that lets us more easily define mappings specific
458           -- for LSP related items. It sets the mode, buffer and description for us each time.
459           local map = function(keys, func, desc)
460             vim.keymap.set('n', keys, func, { buffer = event.buf, desc = 'LSP: ' .. desc })
461           end
462 
463           -- Jump to the definition of the word under your cursor.
464           --  This is where a variable was first declared, or where a function is defined, etc.
465           --  To jump back, press <C-t>.
466           map('gd', require('telescope.builtin').lsp_definitions, '[G]oto [D]efinition')
467 
468           -- Find references for the word under your cursor.
469           map('gr', require('telescope.builtin').lsp_references, '[G]oto [R]eferences')
470 
471           -- Jump to the implementation of the word under your cursor.
472           --  Useful when your language has ways of declaring types without an actual implementation.
473           map('gI', require('telescope.builtin').lsp_implementations, '[G]oto [I]mplementation')
474 
475           -- Jump to the type of the word under your cursor.
476           --  Useful when you're not sure what type a variable is and you want to see
477           --  the definition of its *type*, not where it was *defined*.
478           map('<leader>D', require('telescope.builtin').lsp_type_definitions, 'Type [D]efinition')
479 
480           -- Fuzzy find all the symbols in your current document.
481           --  Symbols are things like variables, functions, types, etc.
482           map('<leader>ds', require('telescope.builtin').lsp_document_symbols, '[D]ocument [S]ymbols')
483 
484           -- Fuzzy find all the symbols in your current workspace.
485           --  Similar to document symbols, except searches over your entire project.
486           map('<leader>ws', require('telescope.builtin').lsp_dynamic_workspace_symbols, '[W]orkspace [S]ymbols')
487 
488           -- Rename the variable under your cursor.
489           --  Most Language Servers support renaming across files, etc.
490           map('<leader>rn', vim.lsp.buf.rename, '[R]e[n]ame')
491 
492           -- Execute a code action, usually your cursor needs to be on top of an error
493           -- or a suggestion from your LSP for this to activate.
494           map('<leader>ca', vim.lsp.buf.code_action, '[C]ode [A]ction')
495 
496           -- Opens a popup that displays documentation about the word under your cursor
497           --  See `:help K` for why this keymap.
498           map('K', vim.lsp.buf.hover, 'Hover Documentation')
499 
500           -- WARN: This is not Goto Definition, this is Goto Declaration.
501           --  For example, in C this would take you to the header.
502           map('gD', vim.lsp.buf.declaration, '[G]oto [D]eclaration')
503 
504           -- The following two autocommands are used to highlight references of the
505           -- word under your cursor when your cursor rests there for a little while.
506           --    See `:help CursorHold` for information about when this is executed
507           --
508           -- When you move your cursor, the highlights will be cleared (the second autocommand).
509           local client = vim.lsp.get_client_by_id(event.data.client_id)
510           if client and client.server_capabilities.documentHighlightProvider then
511             vim.api.nvim_create_autocmd({ 'CursorHold', 'CursorHoldI' }, {
512               buffer = event.buf,
513               callback = vim.lsp.buf.document_highlight,
514             })
515 
516             vim.api.nvim_create_autocmd({ 'CursorMoved', 'CursorMovedI' }, {
517               buffer = event.buf,
518               callback = vim.lsp.buf.clear_references,
519             })
520           end
521         end,
522       })
523 
524       -- LSP servers and clients are able to communicate to each other what features they support.
525       --  By default, Neovim doesn't support everything that is in the LSP specification.
526       --  When you add nvim-cmp, luasnip, etc. Neovim now has *more* capabilities.
527       --  So, we create new capabilities with nvim cmp, and then broadcast that to the servers.
528       local capabilities = vim.lsp.protocol.make_client_capabilities()
529       capabilities = vim.tbl_deep_extend('force', capabilities, require('cmp_nvim_lsp').default_capabilities())
530 
531       -- Enable the following language servers
532       --  Feel free to add/remove any LSPs that you want here. They will automatically be installed.
533       --
534       --  Add any additional override configuration in the following tables. Available keys are:
535       --  - cmd (table): Override the default command used to start the server
536       --  - filetypes (table): Override the default list of associated filetypes for the server
537       --  - capabilities (table): Override fields in capabilities. Can be used to disable certain LSP features.
